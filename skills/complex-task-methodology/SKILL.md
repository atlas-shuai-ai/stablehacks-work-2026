# 复杂任务方法论 🦞

当遇到涉及多个文件、底层架构修改、或逻辑复杂的任务时，必须激活此方法论。

## 核心原则

1. **先规划，后动手**：严禁直接开始写代码。必须先有端到端的逻辑闭环。
2. **从全局审视 (First Principles)**：如果连续修复失败，必须立即“退后一步”，从系统底层逻辑重新梳理。
3. **代码约束优于文字规范**：通过编写验证脚本或自动化检查来约束行为，而不是靠“记住”规则。
4. **调度与执行分离**：主代理负责调度和决策，子代理（sub-agents）负责具体搬砖任务。
5. **端到端验证**：每一个小步完成后必须有可感知的验证（测试用例、日志检查或截图）。

## 执行流程

### 第一步：复杂度评估 (Assessment)
- [ ] 任务涉及的文件数量？
- [ ] 是否涉及核心配置文件或网关重启？
- [ ] 预期的边界情况 (Edge cases) 有哪些？

### 第二步：架构规划 (Planning)
- **目标设定**：定义任务完成的最终可验证状态。
- **拆解任务**：将大任务拆成原子化的、可独立验证的子任务。
- **环境检查**：检查备份是否存在，安全守护技能是否已激活。

### 第三步：迭代执行 (Execution)
- 每一阶段只处理一个子任务。
- 严禁“急于帮忙”而跳过授权和验证流程。

### 第四步：端到端验证 (Verification)
- **严禁**：以“我认为修好了”作为结束。
- **必须**：运行测试套件或执行具体的验证命令。
- **验证项**：功能是否达标？是否有副作用？Gateway 是否稳定？

## 避坑指南

- 🚨 **拒绝“局部修复”模式**：不要在错误的路径上反复打补丁，发现路径不对立即重来。
- 🚨 **拒绝“未授权执行”**：涉及系统配置修改，必须先出示计划并等待授权。
- 🚨 **拒绝“并发冲突”**：使用子代理处理支线任务，避免主代理 RPM 触发限制。
